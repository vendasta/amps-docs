## QA: Onboard Call {#qa-onboard-call}

     1. The member joining the onboard call will be assigned the task _of Reviewing Assets_ to prepare for the call before it is booked. Review the notes from the Team lead in the _Fulfillment Audit_ task and note down any additional assets you need that will be discussed in the onboarding call.
     2. Close the task with notes
     3. Follow the _Onboard Checklist_ during the call. 
        1. When joining the onboard/consultation call, ensure that the call is recorded with a transcript. Enable Gemini to take notes for automatic summarization. Share the summary documentation link with the client engagement and the fulfillment team (website designer and content writers). Instructions on how to enable Gemini notes can be found [here](https://support.google.com/meet/answer/14754931?sjid=18296887865580154859-NC).  
**For PRBS the CSS will join the call on their own and leave notes in the _Onboarding Call & Summary_ task.**

- **Onboarding Checklist** \- ensure all these points are mentioned in the onboarding call
       1. **🕒 Turnaround Times:**
          1. The turnaround time for up to 10 pages is **5** business days
          2. Additional add-ons, like additional pages or eCommerce will require additional time. An updated timeline will be provided after the onboarding process is complete.
       2. ✏️**Revisions:**
          1. Two major rounds of revisions are included at no additional cost.
          2. Additional revisions beyond the two rounds may impact timelines and scope.
          3. We can continue to tweak small revisions after the site is complete or live to ensure you are fully satisfied.
       3. **🔒 Backend Access:**
          1. We limit access to the site until we have completed the website and set the site live on your custom domain. At this time, please let us know if you’d like edit access to your site and we can update your permissions.
       4. **📅 Due Dates & Edit Management:**
          1. Once a due date is confirmed, no new pins should be submitted in Markup.
          2. Any new edits submitted via email after that point may result in a new due date, depending on the complexity of the request.
    2. Summarize all notes in the onboarding call and summary task, and confirm/communicate the due date.

## Designer: Building Your Pages {#designer-building-your-pages}
     1. Create duplicate copies of the original template pages before starting the build and save them to the library before deleting, so that the template sections can be used during the QA phase if needed or used later if the client/partner wants to use a specific section.
     2. Change the names of the specific pages to what is needed for the navigation. Also, ensure you add the Meta Titles/Description before you start editing the pages.
        1. Ensure that the page slugs have been updated correctly. For example, a Gallery page that has been built using the template page named “Projects” should have the slug as “/gallery”(not “/projects”). 
     3. Try to use the Site Theme Customizer for as many branding choices as possible. The same is true for the Text modules - try to use the widget customizations rather than inline CSS. IE - use the default 
     4. With each page, you should check the mobile versions so that you can ensure that any mobile irregularities aren’t copied from one page to another.
     5. As a note, you should be checking off all of the QA Error Log Sheet’s points in the Fulfilment Log tab. Talk to your team lead about the current version.

## Designer/QA: Completing Revisions {#designerqa-completing-revisions}
     1. **Developer:** When completing revisions on the Markup, reply to the revision with “Done”. **DO NOT resolve the revision.**
     2. **Developer:** If a revision can’t be completed, reply with: the reason the revision can’t be completed and what is needed to complete the revision. DO NOT mention pricing or hourly charges. Also, leave a note in VTM for the revisions that couldn’t be completed.
        1. **If you need clarification on an edit, reply with what you need clarification on in Markup**
        2. **Before you say a revision can’t be completed, check with the QA first to confirm**
        3. To retain white label, access the Markup link through an Incognito window. 
           1. When you submit your first reply, you will be prompted to enter a name. Enter your initials, for example, TQ. You do not need to enter an email.
           2. Click “Post as Guest” button to submit your reply  
![](/img/sop-website-services/accelerated-templated-website-build-duda/26.png)
     3. **QA:** Resolve the revisions that have been completed in Markup once you have done a quality check on them. Review the reason why a revision couldn’t be completed and note it in VTM if the reason is valid. DO NOT resolve a revision that hasn’t been completed; instead, make sure a reply is added as to why it couldn’t be completed.
## QA: Launching Website {#qa-launching-website}
     1. Connect a domain - Publish the site on Duda with a custom domain. Ensure to add “www” before the domain name. For example, a domain name “[example.com](http://example.com)” should be entered as “[www.example.com](http://www.example.com)”.
     2. Update DNS record - Visit the domain registrar website and add the following records. Any pre-existing duplicate record should be deleted. [See details here](https://support.duda.co/hc/en-us/articles/26520132463383-Make-Your-Site-Live-Initial-Steps-and-DNS-Record-Setup).

**CNAME**|  _Host: www_ _Points to:_[_s.multiscreensite.com_](http://s.multiscreensite.com)  
---|---  
**A Records**| **First A Record:** Host/Alias: @Target/Destination: 35.172.94.1**Second A Record:** Host/Alias: @Target/Destination: 100.24.208.97  
---|---  
  
- Configure SSL certificates - Go to SEO & Settings > Site SSL > Generate Certificate. This process may take a few hours.

![](/img/sop-website-services/accelerated-templated-website-build-duda/27.png)

- Once the site is live the website will be transitioned to Website Support by CSS.

## Designer/QA: Annual Website Rebuilds {#designerqa-annual-website-rebuilds}

Annual website rebuild should follow the standard website build and QA process. 
     1. **Client Communication** \- All annual website redesigns need to have an onboarding call attended by a designer or QA, similar to the standard website design OB call.
     2. **VTM task Assignment** \- A website build task and QA task need to be created and assigned to the designer and QA team member. 
     3. **QA Error Log** \- A rebuild website task should follow the standard process of updating the fulfillment and QA error log.
     4. **Timeline** \- Annual redesign should follow the standard website design timeline. The designer will decide a due date for a website with 20+ pages. 

  

_If you have questions, contact MS Cohort Developers._
